Generate reject codes from Core

Core's REJECT_* codes are free-standing `static const unsigned char`
declarations in src/consensus/validation.h, not an enum, so extract_opcodes.py
doesn't see them at all. This adds extract_constants.py for that shape.

The prefix filter is required rather than optional. Without it the extractor
would pick up every constant in the header -- including unrelated ones whose
meaning nobody has checked -- and a table of "constants we happened to find"
is not a specification of anything. An empty match exits 2 rather than
emitting an empty table.

gen_constants_c.py is deliberately generic (--name, --title) rather than
reject-specific. A second generator that is this one with s/reject/foo/ is how
generators drift apart.

Enum members keep the family in the identifier: DOGECOIN_REJECT_INVALID, not
DOGECOIN_INVALID. Stripping the prefix loses which family a code belongs to and
collides with any other constant family that has an INVALID.

Value collisions -- two constants sharing a byte -- are documented in the
generated header rather than silently resolved, since a reverse lookup has to
pick one and picking silently is a guess.

This completes Phase 2: provenance, the height-indexed C API, checkpoints,
deployments, opcodes, and reject codes are all derived from Core rather than
hand-transcribed.
