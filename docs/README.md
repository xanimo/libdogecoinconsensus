# docs/

`flattened_v1_spec.json` — a spec that is **wrong on purpose**, kept as
evidence, not as a reference.

It **flattens Core's consensus fork tree**: one consensus regime per network,
`CMainParams.nPowTargetTimespan = 14400` (the pre-Digishield value), and no
post-145000 regimes at all. Real Core mainnet has three:
`consensus@0 → digishield@145000 → auxpow@371337`.

**A client generated from this spec would hard-fork itself off mainnet at block
145,000.**

## Where it comes from

Generated by the *current* extractor from
[`../sample_chainparams.cpp`](../sample_chainparams.cpp), a hand-written flat
mirror of Core's structure that carries no tree wiring — no `pConsensusRoot`,
no `pLeft`/`pRight`, no `digishieldConsensus`.

```bash
./extract_chainparams.py sample_chainparams.cpp -o docs/flattened_v1_spec.json
```

It is **not** real Core output, and it is not a byte-for-byte copy of the
original v1 artifact — that was never committed and is not recoverable. It is a
faithful reproduction of the same failure: v1 flattened real Core the same way
this sample is flat by construction. Same wrong shape, same wrong values, same
fork. It has one property the lost original didn't: it regenerates from files in
this repo.

## Why keep it

It looks correct. It parses. Every value in it is a real value that appears in
`chainparams.cpp`. It is also a fork. That is what *confidently wrong* means,
and it is the entire failure class this repo exists to eliminate. A wrong
artifact you can open and read is worth more than a paragraph claiming one
existed.

## What it demonstrates about the checks

- `gen_consensus_c.py` **refuses it** — the field-coverage gate rejects
  `nAuxpowStartHeight`, a field the generator doesn't account for. Exit 1.
- `verify_selection.py` **passes it** — and that is correct. It proves the
  argmax reduction matches the BST walk *for the tree in the spec*. Here the
  tree is one node, so the proof is trivially true and tells you nothing: the
  flattening already happened upstream, during extraction. A proof about a spec
  cannot detect what the spec lost before it arrived.

Green from one check is not green. That asymmetry is the most useful thing in
this directory.

## Provenance

`core_commit` is `null`, correctly: this did not come from a Core checkout, so
there is no Core commit to cite. An earlier version of the extractor would have
walked up from the file, found *this* repo's `.git`, and stamped its HEAD as
`core_commit` — a fabricated pin, which is worse than no pin. See the commit
*"Refuse to stamp core_commit when the source isn't Core"*.

`file_sha256` is still exact: it identifies the bytes that were parsed, which is
the whole reason the sha256 rather than the commit is the identity.
